# S3 Backend — Corpus Backup & Restore

## Goal

Validate vykar backup and restore correctness on the S3 backend using the smaller remote corpus.

## Scope

- **Backend**: `s3` (credentials and endpoint from `~/vykar.sample.yaml`)
- **Source dataset**: `~/corpus-remote`
- **Verification**: restored tree matches source tree exactly

## Prerequisites

1. Create config from `~/vykar.sample.yaml` with S3 repo definition
2. `export VYKAR_PASSPHRASE=123`
3. Ensure `rclone` is configured for S3 cleanup (use credentials from sample config)
4. Seed an **incompressible** large file so the multipart-upload path executes.
   The default pack target starts at 32 MiB and the multipart threshold is 16 MiB,
   so a ~64 MiB random (incompressible) file guarantees at least one pack over the
   threshold:
   ```bash
   head -c 67108864 /dev/urandom > ~/corpus-remote/large-incompressible.bin
   ```
   Random bytes do not dedup or compress, so the resulting pack stays large and is
   uploaded via `CreateMultipartUpload` → N×`UploadPart` → `CompleteMultipartUpload`
   rather than a single PUT. Without this, the small remote corpus may never produce
   a >16 MiB object and the multipart path goes untested.

## Remote Cleanup (before each run)

Delete S3 repo contents between runs:
```bash
rclone delete <s3_remote:path> --rmdirs
```
- Do NOT use `rclone purge` (may fail with 403 on restricted buckets)
- If bucket root delete is denied, stick with content-only deletion

## Test Procedure

1. Delete repo from previous runs (best effort):
   ```bash
   vykar --config <config> delete -R s3 --yes-delete-this-repo || true
   ```
2. Initialize repo:
   ```bash
   vykar --config <config> init -R s3
   ```
3. Run backup:
   ```bash
   vykar --config <config> backup -R s3 -l remote-corpus ~/corpus-remote
   ```
4. Confirm snapshot:
   ```bash
   vykar --config <config> list -R s3 --last 3
   ```
5. Capture latest snapshot ID.
6. Restore to empty temp directory:
   ```bash
   vykar --config <config> restore -R s3 <snapshot_id> <restore_dir>
   ```
7. Integrity check:
   ```bash
   vykar --config <config> check -R s3
   ```
8. Delete the tested snapshot:
   ```bash
   vykar --config <config> snapshot delete -R s3 <snapshot_id>
   ```
9. Compact repository packs:
   ```bash
   vykar --config <config> compact -R s3
   ```

## Validation

1. Snapshot exists for label `remote-corpus`
2. Restore completes successfully
3. `diff -qr --no-dereference ~/corpus-remote <restore_dir>` reports no differences
4. The incompressible `large-incompressible.bin` restores **byte-identical**
   (confirms the multipart upload + restore round-trips correctly):
   ```bash
   cmp ~/corpus-remote/large-incompressible.bin <restore_dir>/large-incompressible.bin
   ```
5. `vykar snapshot ... delete <snapshot_id>` exits 0
6. `vykar compact` exits 0
7. Optional: SHA256 manifest comparison; optionally confirm the multipart path ran
   by checking verbose/debug logs for `CREATE_MPU` / `COMPLETE_MPU` activity

## Failure Cases to Record

- S3 auth or endpoint errors
- Partial upload or timeout during backup
- Very slow backups on large corpora (use bounded smoke sources when needed)
- Restore mismatch vs source
- `vykar check` failures
- `vykar snapshot delete` or `vykar compact` failures

## Cleanup

1. Remove restore temp directory
2. Remove the seeded `~/corpus-remote/large-incompressible.bin`
3. Clean S3 path with `rclone delete --rmdirs` unless keeping for debugging
